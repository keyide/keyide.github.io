jsonp ：
JSONP(JSON with Padding)是一个非官方的协议，它允许在服务器端集成Script tags返回至客户端，通过javascript callback的形式实现跨域访问

同源策略的限制

  • 同源策略阻止从一个域上加载的脚本获取或操作另一个域上的文档属性，XmlHttpRequest 只允许请求当前源（域名、协议、端口）的资源
  • 有一些标签具备了跨域资源请求的能力
  • 为了实现跨域请求，可以通过 script 标签实现跨域请求，然后在服务端输出 JSON 数据并执行回调函数，从而解决了跨域的数据请求，但通过 src 请求数据的方式都是 get 方式
  • JSONP 是一个非官方的协议，它允许在服务器端集成 Script tags 返回至客户端，通过 javascript callback的形式实现跨域访问
  • 把后台返回的内容引入 script 进来，不管后台返回的是什么类型都会解析成js并执行
  • 但是这也产生了问题
  • 维护问题：需要前后两端同时进行维护
    • 解决：callback 回调函数 -> success 函数
    • success 在前端中声明，然后把 callback 的函数名传到后台，后台使用
    • 这样后台中将需要函数处理的数据函数拼接在 json 中返回
    • 前端在解析的时候，将直接将之解释成 js，页面中已经声明函数，直接调用
  • 怎么传数据：
    • 因为是get请求 所以说请求时只能在地址栏拼接数据
    • 复杂数据响应困难，所以将响应数据都转化为 json 格式
  • 怎么做到即时调用
    • 解决：动态创建 script 标签

jQuery 中的 jsonp

  • jQuery 中的 jsonp 请求已经封装的 $.ajax() 中

jsonp 的优缺点

  • jsonp 的优点
    • 比XML轻了很多，没有那么多冗余的东西
    • JSON也是具有很好的可读性的，但是通常返回的都是压缩过后的。不像XML这样的浏览器可以直接显示，浏览器对于-JSON的格式化的显示就需要借助一些插件了
    • 在JavaScript中处理JSON很简单
    • 其他语言例如PHP对于JSON的支持也不错
  • jsonp 的缺点
    • JSON在服务端语言的支持不像XML那么广泛，不过JSON.org上提供很多语言的库
    • 如果你使用eval()来解析的话，会容易出现安全问题

最后

  • JSONP 是构建 mashup 的强大技术，但不幸的是，它有一些缺陷，在提交开发资源之前必须认真考虑它们。

  • 第一，也是最重要的一点，没有关于 JSONP 调用的错误处理。如果动态脚本插入有效，就执行调用；如果无效，就静默失败。失败是没有任何提示的。例如，不能从服务器捕捉到 404 错误，也不能取消或重新开始请求。不过，等待一段时间还没有响应的话，就不用理它了。（未来的 jQuery 版本可能有终止 JSONP 请求的特性）。

  • JSONP 的另一个主要缺陷是被不信任的服务使用时会很危险。因为 JSONP 服务返回打包在函数调用中的 JSON 响应，而函数调用是由浏览器执行的，这使宿主 Web 应用程序更容易受到各类攻击。如果打算使用 JSONP 服务，了解它能造成的威胁非常重要
